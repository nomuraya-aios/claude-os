# REVIEW-PLAN — LLMファーストドキュメンテーション整備システム

**作成日**: 2026-04-15 22:20 JST

## Haiku セルフレビュー観点（/impl で実施）

| # | 観点 | チェック方法 | 合否 |
|---|------|------------|------|
| 1 | 完了条件を全て満たしているか | SPEC.md の完了条件チェックリストを順に確認 | - |
| 2 | doc-llm-first.md が4軸を網羅しているか | 判断可能・実装突合・更新トリガー・陳腐化サインの4項目が全てあるか | - |
| 3 | /doc-review が引数なしで動作するか | 引数省略時にカレントディレクトリの .md を列挙する分岐があるか | - |
| 4 | feedback.jsonl 投入に scope フィールドがあるか | jq コマンドに `scope: "documentation"` が含まれているか | - |
| 5 | agent-doc-auditor.sh が LLM を呼ばないか | oh-dispatch・claude・curl APIの呼び出しがないか grep で確認 | - |
| 6 | deprecated-terms.txt が外部ファイルで読まれているか | スクリプト内にハードコードされていないか確認 | - |
| 7 | --dry-run フラグが機能するか | `--dry-run` 実行時に feedback.jsonl への書き込みが起きないか | - |
| 8 | トークン爆発リスクがないか | /doc-review が1ファイル1回評価で完結するか、ループがないか | - |
| 9 | manifest.yaml の構造が既存エントリと一致しているか | 既存エントリと同じキー（name/version/description/install_dest/source_dir/trigger/deps）が揃っているか | - |
| 10 | センシティブ情報が含まれていないか | APIキー・パスワードのハードコードがないか | - |

## Sonnet 最終レビュー観点（/final-review で実施）

| # | 観点 | 判定基準 |
|---|------|---------|
| 1 | 設計意図との乖離 | SPEC.md の「LLMが使える状態を継続的に保つ」という目的に沿っているか |
| 2 | 4軸チェックリストの品質 | 軸の定義が曖昧でLLMがブレる余地がないか。NG判定基準が具体的か |
| 3 | doc-llm-first.md が実際にCLAUDE.mdから読まれるか | core-behaviors.md の @ 参照が正しく追加されているか |
| 4 | agent-doc-auditor の検出精度 | 誤検知（存在するファイルを「乖離」と判定）が起きない設計か |
| 5 | feedback.jsonl との統合 | scope フィールドが既存のagent-system-improverパイプラインと互換か |
| 6 | v3-agent-roles.md への追記が既存フォーマットと一致しているか | テーブルのカラム構造（エージェント/責務/実行タイミング）が合っているか |
| 7 | セルフレビューの見落とし | Haiku が「問題なし」としたが実は問題がある箇所を独立評価 |

## 改修判定基準

- Haiku セルフレビューで「要確認」が0件 → Sonnet は最終確認のみ
- 「要確認」が1〜2件 → Sonnet が改修して即クローズ
- 「要確認」が3件以上 → SPEC.md の実装方針を見直す
