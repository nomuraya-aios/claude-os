# LLM自律実行ループの禁止

**Why:** スクリプトがLLMをループ内で繰り返し呼び出すと、
judgeが全件passを返し続けるケースでループが止まらなくなる。
さらにコンパクション後にClaude Codeがコンテキストを失い
「まだ未処理がある」と誤認して再実行するループが発生する。

**Rule:** 以下のパターンを持つスクリプト・ツールは作ってはならない。

## 禁止パターン

| パターン | 具体例 |
|---------|--------|
| LLMをループ内で繰り返し呼び出す | `for file in files; do claude -p ...; done` |
| 「処理済みを再処理する」フラグ | `--reprocess`, `--retry-all`, `--force-rerun` 等 |
| LLMがscore/判定を出して「合格→続行」を繰り返す自律ループ | judge → score2 → 次の件へ → judge → ... |
| `--limit N` だけで上限を設けたつもりになっている | コンパクション後に再実行されると上限がリセットされる |
| 成果物をcommit/pushしてからさらに次の処理を続ける無限改善ループ | commit → 「残りN件」→ commit → ... |

## なぜ止まらなくなるか

1. Claude Codeがスクリプトを開発・実行するセッションが長くなる
2. コンパクション発生 → コンテキストが薄くなる
3. コンパクション後のClaude Codeは「まだ未処理がある」と誤認する
4. `--reprocess` フラグがあると済んだものを何度でも再処理できる
5. judgeが甘いと全件合格 → 際限なく続く

## 代替設計

- **蓄積と実行を分離する**: ファイルへの情報蓄積はOK。実行は人間が判断してから手動で行う
- **1回限りのバッチ**: 実行済みフラグを厳密に管理し、再実行フラグは作らない
- **LLM呼び出しは1スクリプト1回まで**: ループが必要な場合はシェルスクリプト側でループし、LLMは1件ずつ呼び出さず一括でまとめて渡す

## チェック方法

スクリプトにLLM呼び出しを追加するとき:
- ループの中に `claude`、LLM API呼び出しが入っていないか
- `--reprocess` 系のフラグを追加しようとしていないか
- 「合格したら次へ」の自律判断ループになっていないか
- コンパクション後に同じスクリプトが再実行されたらどうなるかを考えたか
