# ラリー削減エンジニアリング（運用問題の根本予防）

**Why:** 2026年4月のセッション分析で、全セッションの45%が「前に作ったものの修正」に費やされていた。moltbookでは10分間に9コミットの修正ラリーが発生（heartbeat zombie）。トークンの約半分が運用修正に消費されている。

**How to apply:** 実装時に以下の3ゲートを通過させる。ゲートを通過しない実装はpushしない。

---

## Gate 1: 実行環境テスト（push前必須）

**問題:** ローカルmacOSで動いてもlaunchd/Docker/cronで壊れるパターンが最頻出。

**ルール:** 非インタラクティブ環境で動くスクリプトは、push前に以下を確認する。

```bash
# launchd で動くスクリプト → env -i で環境変数なし実行
env -i HOME="$HOME" PATH="/usr/bin:/bin:/usr/sbin:/sbin:/opt/homebrew/bin" \
  bash script.sh

# Docker で動くスクリプト → bash --posix で POSIX 互換確認
bash --posix -n script.sh  # 文法チェック

# 確認すべき落とし穴（moltbook実績）:
# - local 変数を関数外で使用 → bash OK, sh NG
# - BSD mktemp vs GNU mktemp（-t オプションの意味が違う）
# - wc -l の出力に空白が入る（macOS）
# - pipefail が設定されていない
```

**チェックリスト（3項目、暗記レベル）:**
- [ ] `set -euo pipefail` がファイル先頭にあるか
- [ ] `local` は関数内でのみ使っているか
- [ ] 環境変数なしで実行して動くか

---

## Gate 2: フォールバック副作用テスト

**問題:** LLMプロバイダー枯渇・投稿ロジック崩壊で「正常系だけ動く」状態で本番投入 → 異常系で壊れる。

**ルール:** フォールバックチェーンを含むコードは、各段階の失敗をシミュレートしてからpush。

```
テスト設計の問い:
1. 「1段目が失敗したら2段目に行くか」 → 1段目のURLを壊して実行
2. 「全段が失敗したらどうなるか」 → 全プロバイダーをオフラインにして実行
3. 「リトライが無限ループしないか」 → 上限に到達させて確認
```

**やってはいけない設計（moltbook実績）:**
- 静的フォールバック → リセット → フォールバック → リセット の無限ループ
- サーキットブレーカー内にforce_resetを入れる
- 「合格したら次へ」の自律判断ループ（LLM自律実行ループ禁止ルールと重複）

**代替設計:** フォールバックは最大2段。3段以上必要なら設計がおかしい。

---

## Gate 3: 1ラリー完結の自律調査

**問題:** AIが「ユーザーでないと分からない」と誤判定して確認 → ユーザーが「それはAIで調べられる」と指摘 → 1ラリーが2-3に膨張。

**ルール:** 質問する前に以下を全て実行する。実行してなお分からなければ初めて聞く。

```
調査順序（上から順に）:
1. ~/.openclaw/ → キー・設定・プロバイダー情報
2. ~/.config/ → アプリ固有設定
3. ~/.claude/ → フック・スキル・ルール
4. ~/workspace-ai/ → リポジトリ・スクリプト
5. git log → 過去の変更・判断
6. gh issue list → 既知の問題
7. grep → コードベース内の文字列検索
```

**禁止パターン:**
- 「○○のURLを教えてください」→ grepで探す
- 「○○はどこにありますか？」→ findで探す
- 「○○の設定はどうなっていますか？」→ catで読む

## 運用コスト意識チェック（実装完了時に自問）

実装が完了したとき、以下を自問する:

1. **この実装は修復スクリプトを必要とするか？** → 必要なら設計が複雑すぎる
2. **壊れたとき、次のセッションで直すのに何ラリー必要か？** → 3以上なら設計を簡素化
3. **watchdog/monitoring が必要か？** → 必要なら「壊れる前提の設計」になっている

**目標:** 修復スクリプト0本の設計。壊れないものは直す必要がない。
