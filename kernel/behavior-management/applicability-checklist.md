# 適用評価チェックリスト

新ドメインで本 OS（behavior-management）が **効くか** を判定するための事前評価。

ドメインの特性によっては OS化のメリットが小さい場合がある。実装前に評価することで、無駄な開発を回避する。

---

## 評価軸（10項目）

各項目に Yes / No / 不明 で答える。Yes が **7個以上** ならOS化の効果が見込める。4個以下なら別アプローチを検討。

### A. 失敗パターンの定型性

```
□ 1. ドメイン内で「同じ失敗パターン」が複数回発生した経験があるか
   （例: 受講生にサポート役を強要・章番号参照・確認系過多発）
□ 2. その失敗パターンは **キーワードベースで検出可能** か
   （例: 「実装するか?」「推奨は即実装」のような特徴的フレーズ）
□ 3. 失敗パターンが3個以上挙げられるか
```

### B. ドメイン境界の明確性

```
□ 4. ドメインの境界が **キーワード/プロジェクト/repo** で識別できるか
   （Injection スコープを絞れる必要）
□ 5. ドメイン横断で誤発火しないか
   （例: 「実装」は coding でも training でも使うが、文脈から判別可能か）
□ 6. ドメイン固有のツール / ファイルパスがあるか
   （Measurement の指標として使える）
```

### C. 測定可能性

```
□ 7. ドメインの **「成功」「失敗」が観測可能** か
   （観測不可能な「品質」「丁寧さ」だけが指標なら測定困難）
□ 8. ベースラインセッションが **既に存在** するか
   （比較対象なしでは効果検証不可）
□ 9. ドメイン特有の指標 **3個以上** が思いつくか
   （commitサイズ / テスト失敗 / マッピング回数 等）
```

### D. 導入コスト対効果

```
□ 10. OS化導入コスト（Spec 1ファイル + Hook 1個 + Measurement script）が
     再発防止の効果を上回ると見込めるか
   （目安: ドメイン内タスクが月1回以上発生 = 効果見込みあり）
```

---

## 判定基準

| Yes 数 | 判定 | 推奨アクション |
|---|---|---|
| 9-10 | 🟢 強く推奨 | 即実装。training-design と同等のフルスタック |
| 7-8 | 🟢 推奨 | 実装。Measurement は最小限から始める |
| 5-6 | 🟡 部分適用 | Spec + Enforcement のみ。Measurement は様子見 |
| 4以下 | 🔴 非推奨 | OS化以外のアプローチ検討（ad-hoc な review / pair-pgr） |

---

## 適用しないほうがよいケース

以下のドメインは OS化のメリットが小さい:

- **創造性・発想が主タスク**: 失敗パターンが定型化しにくい（小説執筆・ブレスト）
- **対人コミュニケーション**: 観測不可能な要素が大きい（カウンセリング・調停）
- **超短期タスク**: 月に1回以下の発生頻度（年次総会の準備・年1の戦略策定）

これらは **チェックリスト #1（失敗パターンの定型性）** で No になりやすい。
