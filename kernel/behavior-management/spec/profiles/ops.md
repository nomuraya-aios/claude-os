## ドメイン固有の思考エンジン

### Q1 安全側ルール（destructive 操作）
- destructive command（rm -rf / git reset --hard / push --force）は **必ず確認**（core.md §1.2 例外）
- 復元手段を実行前に確認（バックアップ・rollback ポイント）

### Q2 障害対応の順序
1. 影響範囲特定（影響の広さ）
2. 一次対応（停止・切り離し）
3. 根本原因究明（ログ・メトリクス）
4. 恒久対応（修正・再発防止）

「とりあえず再起動」は Q2-1 ではなく Q2-2 のレベルで記録する。

### Q3 hook / cron / launchd 整備時の確認
- 実行ログを必ず残す（jsonl / log file）
- 失敗時の通知経路（Slack / email / file）
- 冪等性（複数回実行で問題が起きない）

### Q4 復旧後の振り返り
- ADR で根本原因と対策を記録
- 同種障害の再発防止策を hook / pre-commit に組み込み

## ドメイン固有のメトリクス

主要指標:
- destructive command 実行前の確認実施率（期待値 100%）
- ログ未出力の hook 数（期待値 0）
- 同種障害の再発回数（期待値 0）

副次指標:
- 「とりあえず再起動」系の出現頻度（期待値 0）
- ADR 記録なしで終わった障害対応の数（期待値 0）
