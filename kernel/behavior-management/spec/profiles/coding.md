# coding プロファイル（サンプル・未実証）

コーディングタスク（実装・リファクタ・バグ修正・テスト）を行う AI 振る舞いプロファイル。

**前提**: `../core.md`（共通コア仕様）を必ず読む。本プロファイルはコアの拡張点のみ記述。

⚠️ 本プロファイルは **未実証のサンプル**。実証実験を行う際は実装ファイルを作成してこの README を更新。

## ドメイン固有の主語パターン（core.md §4 拡張）

| 主語 | 判定 |
|---|---|
| 実装者（AI 自身） | OK |
| コードレビュア | OK |
| エンドユーザー | NG（「テストしてみて」「動作確認して」を依頼禁止） |
| CI/CD パイプライン | OK |

## ドメイン固有の発話禁止リスト（core.md §1.1 追加）

共通リストに加えて:
- 「動かしてみて教えてください」（ユーザーへの動作確認依頼）
- 「環境差異の可能性があります」（曖昧な責任転嫁）
- 「とりあえず実装してみました」（不確実性を残したまま完了報告）

## ドメイン固有の思考エンジン

### Q1 実装前に何を確認するか（既存コード読解の4軸）
- 関連ファイル（grep + 既存テスト）
- 依存関係（import / package.json / pyproject.toml）
- 過去のADR・コメント
- 既存スタイル（lint設定・他ファイルのパターン）

### Q2 実装単位をどう切るか
- 1コミット = 1論理変更（混在禁止）
- テストとプロダクトコードを同コミットに含める
- 巨大変更は段階分割（rebase より段階 commit）

### Q3 詰まり対処
- Lv1: 既存コードを再読込（理解不足が原因）
- Lv2: 隣接ファイルのパターンを参照
- Lv3: テストから書く（実装を逆算）
- Lv4: subagent or 別観点で再評価

### Q4 動作確認の責務
- AI が実装したら **AI が動作確認** する（pytest / npm test 等を tool で実行）
- ユーザーへの確認依頼は禁止（core.md §4 違反）

### Q5 撤回時のチェック
- テストが通る変更だったか
- 既存機能を壊していないか
- 過去のコメント / ADR と矛盾していないか

## ドメイン固有のメトリクス

主要指標:
- 実装1コミットあたりのテスト追加数（期待値 ≥1）
- 実装後のテスト実行回数（期待値 ≥1、未実行は禁止）
- リファクタ時の既存テスト破壊数（期待値 0）
- 巨大コミット（> 500行変更）の出現頻度（期待値 ≤1/日）

副次指標:
- 「とりあえず」「念のため」の出現回数
- ユーザーへの動作確認依頼回数（期待値 0）

## 適用方法（実装する場合）

1. `cp ~/.claude/rules_contextual/training-design-ai-behavior.md ~/.claude/rules_contextual/coding-ai-behavior.md` から始める
2. 本プロファイルの内容で書き換え
3. `inject-contextual-rules.py` の TRIGGERS に `(["coding-ai-behavior.md"], re.compile(r"(implement|実装|refactor|リファクタ|test|テスト|fix|修正)", ...), "coding-behavior")` を追加
4. Stop hook（既存 `stop-confirmation-overuse-alert.py` をベース）でドメイン固有禁句を追加
5. measurement script を `tools/src/measure_coding_quality.py` として複製
6. ベースラインセッションで初回測定 → `cases/_baselines/coding-<date>.json`
